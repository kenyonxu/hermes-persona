## User Story

> **作为** hermes-persona 的运维者（主人 Kai.Xu），
> **我希望** 在 `persona-config.json` 中拥有一个集中式的模块开关面板，可以独立启用/禁用每一个注入模块（时间、静态规则、动态规则子通道、随机变化等），并开启 Debug Mode 让知惠在对话中直接报告本轮注入了哪些内容，
> **以便** 我能精确控制知惠的人格注入行为，快速定位问题（如轮数阈值过早触发），不用翻 config 文件就能看清注入引擎的实际运作，以及为未来更多的注入模块提供统一的开关基础设施。

---

## 背景与动机

### 现状问题

1. **开关分散** —— 当前只有 `time.enabled`、`memory.enabled`、`project.enabled` 三个开关，分别埋在各自的配置章节里，缺乏统一入口。
2. **部分模块不可控** —— 静态规则 (`context.rules`)、动态规则 (`dynamic.*`)、随机变化 (`variance.*`) 没有独立开关，想关只能删配置。
3. **排查困难** —— 2026-05-18 发现 `turn_stage` 因 conversation_history 计数污染导致 `after_30` 规则过早触发，主人只能临时将阈值放大 10 倍来缓解。如果有模块开关，直接关掉 `turn_stage` 即可。
4. **扩展无规范** —— 未来加入 Rule-Based 模式切换、Utility 加权决策、主动问候等新模块时，没有统一的开关注册机制。

### 讨论上下文

2026-05-18 周一早晨散步讨论，主人提出四层演进路线：

```
模块开关 → Rule-Based 模式切换 → Utility-Based 模式切换 → 本地小后端
   👈 本次
```

模块开关是所有高级功能的地基——后续的模式切换需要知道「哪些模块在通电」才能做出正确的状态决策。

## 验收标准

### AC-1: 集中式开关配置

`persona-config.json` 顶层新增 `modules` 键，格式如下：

```json
{
  "hermes-persona": {
    "modules": {
      "time": true,
      "static_rules": true,
      "dynamic": {
        "time_slots": true,
        "turn_stage": true,
        "keyword": true
      },
      "variance": true,
      "memory": false,
      "kanban": true,
      "debug": false
    }
  }
}
```

- 每个模块对应一个布尔开关。
- 关闭的模块在 `inject_context()` 中完全跳过，不产生任何输出。
- 格式与现有 `time.enabled` / `memory.enabled` 设计语言一致。

### AC-2: 向后兼容与迁移

- 保留现有 `time.enabled`、`memory.enabled`、`project.enabled` 作为**旧格式支持**。
- 如果 `modules` 键存在，优先使用 `modules`；否则回退读取旧格式。
- 旧格式的读取行为不变——不破坏已部署的配置文件。

### AC-3: 模块注册规范

- 新增模块只需两步：
  1. 在 `persona-config.json` 的 `modules` 下添加新键
  2. 在 `inject_context()` 中对应位置添加 `if modules.xxx:` 守卫
- 模块键名与注入函数一一对应，命名遵循 `snake_case`。
- 在模块注册表中记录所有已注册模块的元信息（名称、描述、默认值、所属阶段）。

### AC-4: 动态规则子通道独立控制

`dynamic.time_slots`、`dynamic.turn_stage`、`dynamic.keyword` 三个子通道各有一个独立开关。关闭任一子通道时，对应的 `_match_*()` 函数不被调用。

这是从本次 bug 中得到的直接教训——`turn_stage` 的过早触发应该能单独关掉，而不影响 `time_slots` 和 `keyword`。

### AC-5: 测试覆盖

- 每个模块开关的 on/off 状态各有一个测试用例。
- 测试验证：开关关闭时，注入上下文中不出现该模块的任何输出。
- 边界测试：`modules` 键缺失时回退旧格式；空 `modules` 对象时优雅降级。

### AC-6: Debug Mode（v1.1 新增）

- `modules.debug` 默认为 `false`（对用户透明）。
- 当 `debug: true` 时，`inject_context()` 在返回的 context 末尾追加一段人类可读的注入摘要，格式如下：

```
🔧 [Debug] 本轮注入:
  ① 🕐 时间已注入
  ② 📜 7条静态规则
  ③ ⚡ time_slots: on / turn_stage: on → after_30 / keyword: off
  ④ 🎲 fox_ears: off / fox_tail: on / metaphor: off
  ⑤ 🧠 已停用
  ⑥ 📋 Ace Music: P0 / Clef: P0 / ...
```

- Debug 摘要**不替代**正常注入——只是在正常人格上下文之后追加。
- Debug 信息的格式由 `_debug_summary()` 函数生成，内容从 `modules` 状态和 `parts` 列表动态提取。
- Debug Mode 只增加一次函数调用和字符串拼接，开启/关闭不影响注入性能。

## 技术设计要点

### 影响范围

| 文件 | 改动类型 |
|------|----------|
| `persona-config.json` | 新增 `modules` 键 |
| `hermes_persona/injector.py` | 为每个注入步骤添加开关守卫 |
| `hermes_persona/dynamic_rules.py` | 子通道开关传入 `_select_dynamic_rules()` |
| `tests/` | 新增模块开关测试 |

### 开关守卫模式

```python
def inject_context(...):
    config = _load_config()
    modules = config.get("modules", {})  # 新格式
    parts = []

    # ① Time
    if _is_enabled(modules, "time", config, "time"):
        parts.append(_time_context(fmt))

    # ② Static rules
    if _is_enabled(modules, "static_rules", config):
        parts.extend(_inject_static_rules(...))

    # ③ Dynamic rules（子通道可控）
    if _has_any_dynamic(modules):  # 任一子通道开启才进入
        parts.extend(_select_dynamic_rules(
            dynamic_cfg, user_message, is_first_turn, turn_count,
            modules=modules.get("dynamic", {})  # 传入子通道开关
        ))
    ...
```

### `_is_enabled()` 辅助函数

```python
def _is_enabled(modules: dict, key: str, config: dict, legacy_key: str = None):
    """优先读 modules.<key>，回退读 config.<legacy_key>.enabled"""
    if modules:
        return modules.get(key, True)  # 默认开启
    if legacy_key:
        return config.get(legacy_key, {}).get("enabled", True)
    return True
```

### 不变量

- 注入顺序不变（Spec D2: Time→Static→Dynamic→Variance→Memory→Kanban）
- 异常处理不变（任何模块内部异常不阻断后续模块）
- Hook 注册不变（`pre_llm_call` / `pre_tool_call` / `post_tool_call` 始终注册）

## 与下一步规划的关系

| 规划层 | 依赖此 US | 关系说明 |
|--------|:---:|------|
| Rule-Based 模式切换 | ✅ 直接依赖 | 需要知道哪些模块在运行才能做 if-then 决策 |
| Utility-Based 模式切换 | ✅ 间接依赖 | 效用计算需要模块状态作为输入特征 |
| 本地小后端 | ✅ 远期依赖 | 后端收集模块状态用于 idle 感知和主动问候 |
