## User Story

> **作为** hermes-persona 的使用者和运维者（主人 Kai.Xu），
> **我希望** 在 `persona-config.json` 中配置一个多维表达向量系统——用关键词累加命中分数、权重和自然衰减，每轮注入一个软提示到知惠的上下文，
> **以便** 知惠的表达底色能平滑地随对话内容漂移（偏技术/偏陪伴/偏未来愿景），而不是被 Rule-Based 开关「啪」地切到某个固定模式。同时，所有向量维度的关键词列表完全由用户控制。

---

## 背景与动机

### 现状问题

1. **模式切换生硬** — 当前 `keyword` 动态规则触发后以固定文本注入（如「🔨 工蜂模式」），效果是「开/关」，无法表达「60% 技术 + 40% 陪伴」的过渡态。
2. **缺乏累积信号** — 对话中多次提到架构和代码，Rule-Based 只能知道「本轮命中了」，不知道「已经连续聊了 8 轮架构」。
3. **衰减缺失** — 命中的模式提示会一直保留在上下文中，即使对话底色已经变了。
4. **用户不可调** — 关键词列表写死在动态规则 JSON 里，用户无法自定义哪些词触发哪些维度。

### 讨论上下文

2026-05-18 周一晚间，主人在一次肠旺面→折扣菜→洗澡的散步中，和我（知惠）一起将表达规则梳理为三个层次：

```
Rule-Based（开关）       → 关键词命中 → 「啪」切工蜂模式     ← 已实现
Fuzzy Utility（指南针）  → 多维分数 → 「今晚底色偏技术」    ← 👈 本次
Pure Utility（导航仪）   → 效用函数 → 计算最优表达式          ← 远景
```

表达向量是 **Fuzzy Utility 层**——它不替 Agent 做选择，只提供软提示。

同时，固定规则与动态规则的边界也在此次讨论中清晰化：

```
固定层（不依赖对话内容）
  🕐 时间感知           → 已实现
  📏 消息长度           → 待实现（纯脚本统计）
  ⏱️ 回复间隔           → 待实现（JSON 暂存 last_reply_at）
  📊 Session turn 数    → 已计算，待注入（轻量信号）
  🎲 随机变奏           → 已实现

动态层（依赖对话内容匹配）
  ⚡ 关键词 → 模式切换   → 已实现（Rule-Based）
  📊 表达向量           → 待实现（Fuzzy Utility）  ← 👈 本次

远景层（不纳入本轮）
  🫂 双向量对比          → 归专用状态追踪模块
```

## 核心概念：表达向量

### 什么是表达向量

一个**多维度、自衰减、用户可控**的软提示系统。每次消息用关键词匹配 N 个维度，按分数规则累加/衰减，每轮将当前向量值注入 LLM 上下文。

```
消息 → 关键词匹配 → [命中+1, 未命中-0.5, ×1] → 累加/衰减 → 注入
```

### 为什么叫「Fuzzy Utility」

| 特性 | 说明 |
|------|------|
| **Utility** | 多维度分数制——不是 0/1，是「工作 60%、陪伴 30%、愿景 10%」 |
| **Fuzzy** | 不强制——LLM 看到分数后**自己判断**怎么调表达底色，不做硬切换 |

它不是 Rule-Based 体系里的另一个可开关模块——它是一个**不同的注入层**，夹在 `dynamic_rules`（模式开关）和 `variance`（随机微调）之间。

## Config 设计

### persona-config.json 追加 section

```json
"expression_vector": {
  "enabled": true,
  "dimensions": {
    "work":     ["代码", "架构", "Bug", "PR", "测试", "重构", "commit", "push", "部署"],
    "future":   ["愿景", "十年后", "梦想", "未来", "规划"],
    "intimacy": ["陪伴", "累了", "一起", "温暖"]
  },
  "score_rules": {
    "work":     [1, -0.5, 1],
    "future":   [1, -1,   1],
    "intimacy": [1, -0.5, 3]
  },
  "reset": "session",
  "storage_path": "~/.hermes/profiles/{profile}/state/expression_vector.json"
}
```

### 字段说明

| 字段 | 类型 | 说明 |
|------|------|------|
| `enabled` | bool | 总开关，关掉 = 不追踪不注入 |
| `dimensions` | dict[str, list[str]] | N 个维度的关键词列表。维度名锁死，关键词完全由用户控制 |
| `score_rules` | dict[str, [int, float, int]] | `[命中得分, 未命中扣分, 权重]` |
| `reset` | "session" \| "daily" \| "none" | 衰减策略 |
| `storage_path` | path | 磁盘暂存路径，`{profile}` 自动替换 |

### score_rules 设计意图

三个数构成一条**自衰减的动力学曲线**：

```
[命中得分, 未命中扣分, 权重]
```

| 维度 | 命中 | 未命中 | 权重 | 行为特征 |
|------|------|--------|------|----------|
| work | +1 | -0.5 | ×1 | 平潮——缓慢累积、缓慢消退 |
| future | +1 | -1 | ×1 | 平衡——不提就自然下沉 |
| intimacy | +1 | -0.5 | ×3 | 温火——高信号、高粘性，一蹭就稳住 |

> **设计原则**：不同维度「不等价」。intimacy 权重 ×3 意味着同样命中一次，信号强度是 work 的 3 倍——因为亲密语言在人类对话中天然具有更高权重。

## 注入格式

用原始计数值，让 LLM 自己判断：

```
📊 [表达向量] work:8 future:3 intimacy:2
```

> 「work 8 future 3 intimacy 2」比「work 67%」更直觉——因为 0 代表「今天没提过」，具有信号意义。

### Session turn 作为轻量信号

Session turn 数保留，与表达向量并列注入：

```
📊 [表达向量] work:3 future:1 intimacy:8 | 第 22 轮
```

表达向量管「聊了什么」，session turn 管「聊了多久」——两条信号互补，LLM 自行权衡。

## 架构定位

```
注入顺序（不变）：

time_context        → ① 🕐 时间感知
static_rules        → ② 📜 静态规则（骨骼）
dynamic_rules       → ③ ⚡ 关键词 → 模式切换（Rule-Based）
expression_vector   → ④ 📊 表达向量（Fuzzy Utility）  ← 👈 新层
variance            → ⑤ 🎲 随机变奏
memory              → ⑥ 🧠 记忆召回
kanban              → ⑦ 📋 看板
debug               → ⑧ 🔧 调试摘要
```

表达向量夹在 `dynamic_rules`（即时命中）和 `variance`（随机微调）之间。模式切换是开关，表达底色是指南针。

## 固定规则补充（纳入本轮）

以下固定层信号与表达向量一同落地：

| 信号 | 数据来源 | 实现方式 | 注入格式 |
|------|----------|----------|----------|
| 📏 消息长度 | `len(user_message)` | `_message_length_hint()` | 「消息较短」 |
| ⏱️ 回复间隔 | JSON 暂存 `last_reply_at` | `_reply_gap_hint()` + 写回 | 「🎵 欢迎回来」 |

两条都是纯脚本统计，不占 LLM token（仅注入短提示）。

## 验收标准

### AC-1: 配置热加载
- [ ] `expression_vector` 配置修改后无需重启，下一轮自动生效

### AC-2: 关键词匹配与累加
- [ ] 消息中的关键词命中对应维度，按 `[命中得分 × 权重]` 累加
- [ ] 未命中维度按 `[未命中扣分 × 权重]` 衰减
- [ ] 分数不跌破 0

### AC-3: 注入行为
- [ ] 每轮将当前向量值注入 LLM 上下文（格式：`📊 [表达向量] work:N future:N intimacy:N`）
- [ ] 注入位置在 `dynamic_rules` 之后、`variance` 之前
- [ ] `enabled: false` 时不注入，但仍可追踪（由 `track_only` 字段控制）

### AC-4: 持久化与衰减
- [ ] 向量值写入磁盘（`storage_path`）
- [ ] 按 `reset` 策略清零（"session" — 新会话清零, "daily" — 每日零点清零, "none" — 不自动清零）
- [ ] 文件不存在时自动创建且正常初始化

### AC-5: 消息长度与回复间隔
- [ ] 消息长度 <50 字符注入「消息较短」提示
- [ ] 回复间隔 >30 分钟注入「🎵 欢迎回来」提示
- [ ] 回复后更新 `last_reply_at` 时间戳

### AC-6: Session turn 注入
- [ ] 与表达向量同一行注入 session turn 数

### AC-7: 用户可控性
- [ ] 用户可在 `dimensions` 中自由增删每维度的关键词
- [ ] 用户可在 `score_rules` 中调整每维度的命中得分/衰减速率/权重
- [ ] 维度数量可增减（默认 3 维，用户可扩展）

### AC-8: 向后兼容
- [ ] 不配置 `expression_vector` 时，插件正常运行（完全可选）
- [ ] 现有 `modules` 开关不影响 `expression_vector`（独立 `enabled` 字段）

## 与 US-001 的关系

US-001（模块化总控开关）的 `modules` 体系管理 Rule-Based 层的注入模块。US-002 的表达向量是**一个新的注入层**，不属于 `modules` 体系——它有自己的 `enabled` 字段。
