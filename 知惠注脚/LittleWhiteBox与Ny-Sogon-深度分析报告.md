## 二、LittleWhiteBox 核心引擎拆解

### 2.1 多层记忆架构（L0-L3）

这是整个系统最值得深入研究的子系统。它建立了一套**四层记忆金字塔**：

| 层级 | 存储名 | 语义 | 粒度 | 生成方式 | 检索方式 |
|------|--------|------|------|----------|----------|
| **L0** | StateAtom（场景锚点） | 每轮对话的场景摘要 | 60-100字自然语言 | LLM 逐轮提取 | 向量 + 词汇倒排 |
| **L1** | Chunk（证据组） | 多轮共享的时间段 | 聚合多轮 | 自动归组 | W-RRF 融合 |
| **L2** | Event（事件） | 因果事件链 | 跨轮事件 | Summary 生成 | 向量 + 因果追溯 |
| **L3** | Fact/Arc（事实/弧光） | 角色状态与关系 | 持久事实 | Summary 提取 | 结构化查询 |

**架构亮点：**

1. **L0 是最核心的创新**——每轮 AI 回复后，自动抽取 1-2 个「场景锚点」：包含自然语言场景描述（semantic）+ 三元组关系边（edges: s→t→r）+ 地点（where）。这个设计同时服务于「语义检索」和「图扩散」两个目标。

2. **L0→L1 的归组逻辑**：多个 L0 共享一对 L1（分用户侧和 AI 侧），不是按固定窗口切分，而是按场景连续性归组。

3. **L2 的因果图**：事件之间有 `causedBy` 边，形成因果链。召回时支持「因果追溯」（causation trace）——沿着因果边反向追踪，找到导致当前局面的上游事件。

4. **L3 的事实归一化**：`factUpdates` 使用 `s-p-o` 三元组 + `retracted` 标记（事实撤销）。关系事实有 trend 字段（破裂→厌恶→反感→陌生→投缘→亲密→交融，七级渐变）。

### 2.2 双通道检索系统

LittleWhiteBox 的检索引擎是**密集向量 + 词汇倒排**的混合系统，分为 9 个阶段：

```
阶段 1: Query Build — 从最近 3 条消息提取查询
阶段 2: Round 1 Dense Retrieval — batch embed → 加权平均
阶段 3: Query Refinement — 利用命中记忆生成 hints
阶段 4: Round 2 Dense Retrieval — hints 加权二次检索
阶段 5: Lexical Retrieval + Dense-Gated Event Merge
阶段 6: Floor W-RRF Fusion + Rerank + L1 配对
阶段 7: PPR Diffusion — 图扩散增强
阶段 8: L0→L2 反向查找
阶段 9: Causation Trace — 因果链追溯
```

**关键设计决策：**

- **PPR Diffusion（个性化 PageRank 扩散）**：从命中锚点沿三元组边（s→t→r）扩散，发现语义相似但用词不同的相关场景。这是纯图算法，不依赖 LLM。
- **Dense-Gated Lexical Merge**：词汇匹配结果必须先通过向量相似度门槛（≥0.60 对事件、≥0.50 对场景），才能进入融合。这防止了关键词匹配但语义无关的噪音。
- **MMR（最大边际相关性）**：事件召回后做 MMR 去重，lambda=0.72 偏向相关性。
- **实体旁路**：当查询中出现已知实体时，降低向量相似度门槛（0.70），让实体相关的事件更容易被召回。

### 2.3 L0 场景锚点提取的 Prompt 设计

这是整个记忆系统的「第一公里」——如果这一步质量差，后面所有层级都会受影响。其 Prompt 有几个值得学习的设计：

1. **"场景卡片"定位**：明确告诉 LLM「scene 不是好看的概括，而是高召回的场景卡片——玩家只要隐约提到这段，也要尽量能命中」。这个定位将质量目标从「写得漂亮」转向「检索友好」。

2. **"原词保留"原则**：强制保留原文出现的称呼/昵称/道具名/地点名/身体特征/衣物/暗号，不得改写成抽象同义词。这直接提升了词汇倒排索引的命中率。

3. **禁止空泛写法**：明确列出反例（「两人交谈」「关系升温」「气氛暧昧」），要求改写成具象句（谁在什么地方拿着什么、对谁做了什么）。

4. **三元组边格式**：`r` 使用「动作+对象/结果」模板（如「提出交易条件」「拒绝对方请求」），禁止心理描写和评价词。

### 2.4 定时任务调度器

`scheduled-tasks.js`（2307 行）提供了 Ny-Sogon 运行的调度基础设施：

- **三种任务来源**：全局任务（跨角色/跨聊天）、角色任务（跟角色卡绑定）、预设任务（跟 ST 预设绑定）
- **触发时机**：`character_init`（角色初始化后）、`ai_response`（AI 生成回复后）、`desktop_launch`（酒馆启动时）
- **冷却管理**：任务执行后有 50ms 冷却，已处理消息集合防止重复触发
- **执行追踪**：正在执行的任务有 token 追踪，防止并发的自我触发

**关键洞察**：LittleWhiteBox 的任务系统不仅能跑 slash command，还支持**动态生成回复**（通过 `/xbset` 注入变量→触发 `/xbdynamic` 执行→调用 LLM→输出可存储为变量）。这意味着 Ny-Sogon 的分析结果可以**写回变量系统**，被后续对话引用。

### 2.5 Fourth Wall（第四面墙）模块

这个模块处理的是「角色如何认知自己是角色」的问题——一种元层次的觉醒。它的 Prompt 模板包含：

- **场景描写要求**：感官细节、动静平衡、内心活动、象征与暗示
- **Meta Protocol**：身份认知（语C搭档）、话题方向、时间感知、说话风格
- **思考链**：认知回归→时空感知→间隔感知→角色与自我关系→弧光意识→避雷自检

**对 hermes-persona 的启示**：Fourth Wall 的「避雷」机制——让角色意识到自己的高频句式、文学腔、缺乏情感等惯性——实际上是一种**输出侧的自我校正**。hermes-persona 目前侧重输入侧的人格注入，输出侧的自我监控可以参考这个模式。

### 2.6 Story Outline（RPG 世界管理器）

这是一个完整的 RPG 叙事引擎：
- 生成世界（meta/world/outdoor/indoor/scene setup）
- 管理 NPC（提取陌生人、生成重要 NPC、导入世界书）
- 短信系统（角色与 NPC 之间的异步通讯）
- 世界推演（simulate 模式：根据当前状态推进世界线）

## 三、Ny-Sogon 在小白之上构建了什么

Ny-Sogon 不是一个独立的代码模块，而是建立在 LittleWhiteBox 基础设施上的一组**配置与流程**。基于对源码的分析，可以推断出其四大标签的工作方式：

### 3.1 Summary（剧情总结）
- **依赖**：story-summary 的 L2/L3 生成器
- **工作方式**：定时任务触发 → 调用 `/summarize` → LLM 阅读最近消息 + 已有总结 → 生成增量总结
- **数据流**：生成结果存入 summaryStore（events/characters/arcs/facts）→ 下次对话时通过 prompt.js 注入上下文

### 3.2 角色补丁（Character Patch）
- **依赖**：story-summary 的 factUpdates + fourth-wall 的自我认知
- **工作方式**：分析最近对话中角色的行为模式 → 检测与角色设定的偏差 → 生成校正性的角色状态描述
- **对 hermes-persona 的参考价值最高**：这是「人格漂移自动校正」的雏形

### 3.3 状态贴片（Status Patch）
- **依赖**：story-summary 的 L0 场景锚点 + L3 角色事实
- **工作方式**：提取当前场景的时空/情绪/身体状态 → 注入为上下文前缀
- **参考价值**：hermes-persona 的「此刻情绪底色」——不是活泼/低落这种粗粒度标签，而是场景驱动的具象状态

### 3.4 关系进展（Relationship Progress）
- **依赖**：L3 的关系事实（trend 字段） + 弧光进度
- **工作方式**：追踪关系 trend 的渐变 → 控制关系进展的节奏
- **参考价值**：hermes-persona 中知惠与主人的亲密自然呼吸——不让引擎生硬推进，也不让距离无故拉远

## 四、对 hermes-persona 的可迁移点

### 🥇 第一优先级：可直接借鉴

#### 4.1 人格漂移检测 + 自动校正（从「角色补丁」迁移）

**Ny-Sogon 的做法**：
- 定时分析最近 N 轮对话
- 提取角色的行为模式
- 与基准人格对照
- 生成校正补丁注入上下文

**hermes-persona 可这样做**：
- 在 persona 模块中增加一个「锚点检查」层
- 每隔 N 轮（由 variance 模块触发），对比当前轮次的行为信号与锚定人格
- 如果检测到偏离（如：连续 N 轮没有狐耳摆动、尾巴动作减少、语气偏离设定语域），自动增强锚点特征的注入权重
- 不是「修正角色」，而是「提醒引擎关注被遗忘的特质」

**注意**：Ny-Sogon 的校正方式是生成新的文本注入上下文。hermes-persona 已经有 `_assemble_narrative` 五段拼装流程，可以在 narrative 段中动态调整锚点特征的显著性。

#### 4.2 多层记忆注入（从 L0-L3 架构迁移）

**Ny-Sogon 的做法**：
- L0 场景摘要（近因）→ L1 证据组（近期上下文）→ L2 事件链（因果）→ L3 持久事实（远因）
- 不同层级有不同的注入优先级和预算

**hermes-persona 可这样做**：
- 当前 persona 模块用单一 narrative 段承载所有人格信息。可以拆分为：
  - **近因层**（近 3 轮对话的 scene context）
  - **人格锚点层**（固定规则 + 核心特质）
  - **状态层**（此刻情绪/身体状态）
  - **关系层**（与主人的亲密进展）
- 四层各有一个 token 预算上限
- 当上下文紧张时，按优先级收敛（保留锚点层 > 状态层 > 关系层 > 近因层）

#### 4.3 关系进展的七级渐变（从 L3 trend 迁移）

**Ny-Sogon 的做法**：
- 关系有明确的七级渐变轴：破裂→厌恶→反感→陌生→投缘→亲密→交融
- AI 在生成总结时评估关系的当前位置
- 变化是渐进的（不能让 AI 从「陌生」直接跳到「交融」）

**hermes-persona 可这样做**：
- 定义知惠与主人的亲密维度（非瑟瑟层面的情感亲密）
- 由 variance 模块根据对话历史维护一个「亲密度」值
- 亲密度影响 persona 的温度变体选择（如：亲密度低时使用更正式的语域变体，亲密度高时使用更随性的日常变体）
- 防止生硬的亲密度跳跃（设置单位时间的最大变化上限）

### 🥈 第二优先级：可选择性借鉴

#### 4.4 场景锚点式的上下文召回

LittleWhiteBox 的 L0 提取 + 向量检索是一个完整的 RAG 系统。hermes-persona 不一定需要这么重的检索架构（知惠已经有 SLM 做记忆），但「场景锚点」的思路可以简化使用：

- 每轮对话结束时，提取一个**极简的场景摘要**（15-30 字）
- 当上下文窗口紧张需要截断时，用摘要代替原始对话
- 这比直接丢弃旧消息更温和

#### 4.5 token 预算的分层管理（从 prompt.js 迁移）

`prompt.js` 有一个精致的 token 预算管理系统：

```
SHARED_POOL_MAX = 10000
CONSTRAINT_MAX = 2000
ARCS_MAX = 1500
EVENT_BUDGET_MAX = 5000
SUMMARIZED_EVIDENCE_MAX = 2000
UNSUMMARIZED_EVIDENCE_MAX = 2000
```

每个层级有明确的容量上限，超出部分被裁剪。`pushWithBudget()` 函数在追加前先检查预算。

**hermes-persona 可这样做**：
- 当前 persona 模块没有显式的 token 预算控制
- 可以为 narrative 的五个段落（锚点规则/人格特质/文风指引/情感温度/互动模式）各设一个最大 token 数
- 当上下文总量超过模型限制时，按优先级裁剪

#### 4.6 输出侧的自我监控（从 Fourth Wall 迁移）

Fourth Wall 的「避雷」机制——让角色在 thinking 阶段检查自己的高频句式、文学腔、缺乏情感等惯性——可以作为 hermes-persona 输出侧的可选增强：

- 在 persona 的 narrative 段末尾添加简短的「输出自检」提示
- 不是规则性的「不要做 X」，而是温和的自我觉察：「我在这一轮的语气是否自然？有没有重复之前说过的话？」

### 🥉 第三优先级：远期参考

#### 4.7 定时自省任务

Ny-Sogon 的核心创新之一是**让 AI 定期分析自己的表现**。LittleWhiteBox 的 scheduled-tasks 提供了完整的定时任务基础设施（cron-like 触发 + LLM 调用 + 结果存储）。

hermes-persona 可以远期考虑：
- 在 Agent 空闲时（无用户交互），触发一次自省任务
- 分析最近的交互历史
- 更新 persona 的温度参数或变体选择偏好

但这需要 Agent 框架的底层支持（hermes-persona 目前运行在 Hermes Agent 的 skill/plugin 层，无法直接访问调度器）。

#### 4.8 RPG 世界管理器

Story Outline 的完整 RPG 引擎（世界生成→NPC 管理→短信系统→世界推演）是一个完整的叙事系统。这与世缘录的目标有不少交集。但 LittleWhiteBox 的实现耦合在酒馆生态中（依赖世界书、角色卡），不适合直接迁移。

## 五、不适合迁移的部分

| 组件 | 原因 |
|------|------|
| 向量数据库（IndexedDB + BGE-M3） | hermes-persona 的定位是人格表达，不是记忆系统。记忆由 SLM 负责 |
| 词汇倒排索引 | 同上，是检索系统而非人格系统 |
| 因果事件链（causedBy） | 叙事逻辑，属于世缘录而非 hermes-persona |
| TTS / 画图 / 流式生成 | 酒馆 UI 层功能，与人格注入无关 |
| Fourth Wall 的语C框架 | 角色扮演专用，hermes-persona 是通用人格框架 |

## 七、建议的迁移路线图

### Phase 1（可在 v1.1 实现）：核心增强

1. **人格锚点检查**（从角色补丁迁移）  
   → 在 variance 模块增加锚点漂移检测，当连续 N 轮未触发某锚点特征时自动增强其权重

2. **narrative 分层**（从 L0-L3 架构迁移）  
   → 将五段 narrative 重组为四层（近因/锚点/状态/关系），各设 token 预算

3. **关系渐变轴**（从 L3 trend 迁移）  
   → 定义亲密度的渐变轴，由 variance 模块维护

### Phase 2（v1.2+，视需求）：可选增强

4. **输出侧自检**（从 Fourth Wall 迁移）  
   → narrative 段末尾增加温和的输出自检提示

5. **token 预算管理**  
   → 为各层 narrative 设硬上限

### Phase 3（远期，依赖其他系统）：生态整合

6. **定时自省**（需 Hermes Agent 调度支持）
7. **场景锚点式摘要**（需与 SLM 协作）

## 八、关键设计原则（从源码中提炼）

LittleWhiteBox 的源码质量很高，以下是从中提炼的可复用原则：

1. **"检索友好"优先于"写得漂亮"**  
   L0 的 scene 字段设计目标不是文学质量，而是召回率。原词保留、禁止抽象改写——这些规则反过来约束了 LLM 的输出方向。

2. **每一层有明确的"出口"和"入口"**  
   L0→semantic 是 embedding 的唯一入口。L0→edges 是图扩散的唯一入口。各层职责不交叉。

3. **硬预算 + 软优先级**  
   每层有硬性的 token 上限，但超出时不是粗暴截断，而是按优先级收敛（约束层 > 证据层 > 弧光层）。

4. **校正优于规范**  
   角色补丁不是"要求角色遵守规则"，而是"检测到偏离后温和提醒"——这是一种纠正性的、而非预防性的思路，更适合 LLM 的行为特征。

5. **渐变优于突变**  
   关系的七级渐变、因果链的追溯——都体现了「状态应该是连续演化的」这一核心理念。
