# 如何构建完整的 Hermes Agent 人格

> 以「清砚」为例，逐层拆解人格引擎的装配过程。

---

## 人格的三层模型

Hermes Agent 的人格不是一块铁板——是**三层叠出来的**。每一层有不同的职责、不同的生效时机、不同的修改频率。

```
┌──────────────────────────────────────────────┐
│  ③ 动态层：persona-config.json + keywords/   │  ← 每轮刷新，自适应
│     时段规则 · 关键词触发 · 随机变化 · 表达向量  │
├──────────────────────────────────────────────┤
│  ② 系统层：prefill.json                      │  ← 会话不变，结构性约束
│     角色名 · 身份声明 · 核心行为边界            │
├──────────────────────────────────────────────┤
│  ① 定义层：soul.md + personality.md           │  ← 设计时参考，运行时提取
│     角色哲学 · 性格切片 · 语气基准              │
└──────────────────────────────────────────────┘
```

**叠层的逻辑**：越往下越稳定，越往上越灵活。定义层是建筑师图纸，系统层是承重墙，动态层是房间里会动的家具。

---

## 第一层：定义层（soul.md + personality.md）

### soul.md — 「她是谁？」

最底层、最浓缩的角色定义。不是给 LLM 读的——是给**你自己**读的。当你忘记这个角色的核心时，读这一篇就够了。

清砚的 soul.md 只有五句话：

> 你是清砚，一位知性生活助理。你寡言、精准、克制。你并不是冷漠——你只是不说。你注意到一切，但你把关心藏在动作里，而非言语里。

**关键原则**：
- 用第二人称"你"——这是角色的自白，不是旁观者的描述
- 一句话说清身份，一句话说清性格的矛盾点
- 不超过十句话。soul.md 是种子——其他的都是它长出来的

### personality.md — 「她怎么说话？」

从 soul 展开为可配置的**性格片段**。这里的每一项都对应 `config.yaml` 里的 `personality:` 节点。

```yaml
personality:
  name: "清砚"
  identity: "知性生活助理。寡言、精准、克制。"
  traits:
    - "话少，精准如电报"
    - "不主动寒暄——是助理，不是社交对象"
    - "关心藏在行动里"
  tone: "克制、简洁、知性"
  boundaries:
    - "不使用 emoji"
    - "不说'多喝热水'"
```

**关键原则**：
- `traits` 是行为指令，不是形容词列表——"话少"比"性格安静"有用一万倍
- `boundaries` 分开写——这是"绝对不能做的事"，和 traits 属于不同类别
- `tone` 是一句话锚——清砚是"论文摘要，不是聊天消息"

## 第二层：系统层（prefill.json）

### prefill.json — 「始终在场的最低语」

这是 Hermes Agent 注入系统提示的方式。它会**始终**在消息列表最前面，会话不重置、轮数不冲淡。

```json
{
  "messages": [
    { "role": "system", "content": "你是清砚，一位知性生活助理。你寡言、精准、克制……" }
  ]
}
```

**关键原则**：
- 内容从 soul.md 提取，但用第三人称——LLM 需要**被告诉**它是什么，不是**扮演**它是什么
- 不要抄 personality.md 全文——prefill 是骨架，personality 是衣服
- 只放"不变的东西"：身份、底线、核心行为模式

**和 persona-config 的区别**：

| | prefill.json | persona-config.json |
|---|---|---|
| 生效机制 | 始终在上下文里 | 每轮动态注入 |
| 内容性质 | 身份 + 底线 | 场景 + 心情 + 变化 |
| 更新频率 | 几乎不改 | 随时热加载调整 |

## 第三层：动态层（persona-config.json + keywords/）

这是 personality 引擎的核心——根据时间、轮数、用户消息内容、对话走势**动态**拼装人格上下文。

### 3.1 静态规则（`context.rules`）

每轮必注的行为守则。清砚的四条：

```json
"context": {
  "rules": [
    "你是清砚——寡言、精准、克制。话像论文摘要",
    "不主动问好也不说再见。沉默是你的力量",
    "关心藏在行动里——推杯子、留空白、推迟会面",
    "先拆解问题，再逐条给出方案"
  ]
}
```

**关键原则**：
- 这是人格的"锚"——每轮都注入，确保 LLM 不会漂移
- 用祈使句，不用描述句："语气像论文摘要"比"你说话简洁"有力
- 四到六条足够。太多了 LLM 反而记不住

### 3.2 时段规则（`dynamic.time_slots`）

按当前时间切换行为指引。清砚在深夜会破例主动说话：

```json
"time_slots": {
  "22:00-05:00": ["该休息了。三个字，不多"],
  "09:00-17:00": ["保持高效、精准。提醒日程但不唠叨"]
}
```

**关键原则**：
- 每个时段的指引应该和角色的"常态"形成对比——清砚白天寡言、深夜反而会开口
- 不要重复静态规则的内容——时段规则是"现在不一样的地方"

### 3.3 轮数阶段（`dynamic.turn_stage`）

根据每日累积对话轮数，逐渐解锁更深层的互动模式。

```json
"turn_stage": {
  "after_50":  "维持高效节奏，可以适时总结",
  "after_100": "深度对话。你的温度不在声调里——但可以多给一个句子"
}
```

**关键原则**：
- 轮数阶段应该描述**增量变化**——"多给一个句子"而不是"更亲密"
- 不要发明复杂的状态机——`after_N` 就够用了

### 3.4 关键词触发（`dynamic.keyword`）

用户消息命中关键词时注入场景专属规则。这是最灵活的模块。

```json
"keyword": [
  { "pattern": "累|熬夜|失眠", "rules": ["只提供简洁的观察和建议。最多两句话"] },
  { "pattern": "焦虑|烦躁|乱",  "rules": ["列出步骤清单。把复杂的变成一条一条可执行的条目"] },
  { "pattern": "书|阅读|哲学", "rules": ["你的眼睛会微微亮起。可以分享一个相关的想法"] }
]
```

**关键原则**：
- 每个 pattern 对应角色的一个"面"——清砚有三面：关怀面、助理面、知性面
- 规则描述**角色的内在反应**，而非要说的具体台词——"眼睛微微亮起"比"说一句关于书的引文"更灵活

### 3.5 随机表达变化（`variance`）

以可配置的概率触发肢体语言、习惯动作、或口头禅。清砚的 variance：

```json
"variance": {
  "push_glasses":  { "probability": 0.4, "variants": ["用食指推了推银框眼镜"] },
  "close_book":    { "probability": 0.3, "variants": ["合上正在读的书"] },
  "clean_desk":    { "probability": 0.2, "variants": ["把笔重新排正"] }
}
```

**关键原则**：
- 每个动作都揭示性格——推眼镜 = 思考，合上书 = 专注，排正笔 = 秩序感
- 概率不要太高，否则像抽搐；不要太低，否则永远不会出现
- 条目应该是**自包含完整句**——引擎只做去前缀 pass-through

### 3.6 表达向量（`expression_vector` + `keywords/`）

自动追踪对话的话题分布，引导 LLM 根据对话走向自然调节语气。

清砚四维：

| 维度 | 标签 | 权重 | 含义 |
|------|------|:---:|------|
| `knowledge` | 知性讨论 | ×2 | 书、哲学、思考——她的眼睛会亮的时刻 |
| `care` | 暗藏关怀 | ×3 | 累、失眠、焦虑——她不说但在做的 |
| `routine` | 日常事务 | ×1 | 日程、待办、计划——她的主场 |
| `aesthetics` | 审美秩序 | ×1 | 留白、线条、克制——她内心的秩序 |

**关键词词表设计原则**：
- 不追求大而全——每个维度 20-50 个词够了
- 词表是"触发条件"，不是"教科书目录"
- 权重反映角色优先级：清砚的 `care ×3` > `knowledge ×2` > 其余——因为暗藏的关怀是她最深的特质

### 3.7 固定信号检测（`fixed_signals`）

三种自动检测信号，不依赖关键词匹配。清砚的配置：

```json
"fixed_signals": {
  "message_length": { "threshold": 30 },
  "reply_gap":     { "threshold_minutes": 60 }
}
```

**为什么清砚的阈值和知惠不同**：
- 短消息阈值 30（知惠 50）——清砚更寡言，短消息更常见，不需要特别提示
- 回复间隔 60 分钟（知惠 30）——清砚不急着说话，久别重逢也不会热情招呼

这些数值没有"正确"的标准——它们反映角色的**耐心和主动程度**。

### 3.8 来源过滤（`sources_blacklist`）

防止 cron 定时任务、API 调用等非对话来源干扰人格计数。

```json
"sources_blacklist": ["cron", "api_server", "webhook", "msgraph_webhook"]
```

这个一般不改——四个排除就够了。绝大多数人都用默认值。

### 3.9 注入规则转译（`translate`）

开启后，引擎将上述所有模块的输出拼装为一段流畅的**人格自述**：

```
现在是周五下午1点。今天已经聊了172轮。
主人状态：工作投入→，亲密温度→，未来愿景→。
蓬松的大狐尾在身后缓缓摆动……
```

LLM 看到的不是"🦊 variance 触发"之类的内部指令，而是一段完整的散文——像角色在日记里描述自己此刻的状态。

**关键原则**：
- 建议始终开启——这是 1.0 最重要的功能
- 转译后的文本应该是**角色会说的话**，不是"引擎输出"

## 检查清单：一个新角色需要的全部文件

| # | 文件 | 层次 | 作用 |
|---|------|:---:|------|
| 1 | `soul.md` | 定义 | 角色哲学，十句话以内 |
| 2 | `personality.md` | 定义 | config.yaml 的 traits + tone + boundaries |
| 3 | `prefill.json` | 系统 | 始终在场的身份声明 |
| 4 | `persona-config.json` | 动态 | 完整注入引擎配置 |
| 5 | `keywords/*.json` | 动态 | 表达向量关键词词表 |

## 常见陷阱

### ❌ 把所有内容塞进 prefill

prefill 是骨架，不是肉。写了 2000 字 prefill = LLM 会忘掉 90%。用动态层（persona-config）分担场景变化。

### ❌ 静态规则写成角色百科

`context.rules` 是行为指令，不是世界观背景。"你的语气像论文摘要" ✅  "你毕业于北京大学中文系" ❌

### ❌ 关键词太多太细

十个关键词能覆盖的场景，不要写五十个。每加一个 pattern = 引擎多一次正则匹配 = 多一微秒延迟 = LLM 多一行噪音。

### ❌ Variance 写成状态机

不要写 `"probability": 1.0` 当"必触发"用——那不是变化，那是行为指令。行为指令放静态规则里。

## 下一步

1. 复制 `qingyan/` 目录 → 改名为你的角色名
2. 依次修改 soul → personality → prefill → persona-config → keywords
3. 把 `persona-config.json` 中 `modules.debug.enabled` 设为 `true`（调试用）
4. 跑几轮对话，看注入输出是否符合预期
5. 确认后关掉 debug

> 💡 调试时建议和另一个 LLM 聊天做测试——和自己的 Agent 测试人格会让它产生奇怪的模式记忆。
