# Persona Craft 架构方案 v0.1

> **设计原则**：零源码侵入，完全基于 Hermes 现有扩展机制（MCP Server + Cron Job + Webhook）实现。

---

## 一、方案概述

随着翁法罗斯角色素材持续增加，直接将数万字 lore 硬编码进 `prefill.json` 会导致三个致命问题：
1. **Token 成本爆炸**：每次对话都携带完整人设，经济性极差。
2. **人格稀释**：信息过载会让模型抓不住「说话的感觉」。
3. **维护困难**：剧情更新（如 3.0 → 3.7 关系描述变化）时，需要全量重写。

**本方案采用「分层 + 动态召回」策略**：
- `prefill.json` 只保留**最精炼的人格内核**（语气、性格标签、核心信念）。
- **海量 lore** 由 MCP Server 按需检索，只在聊到相关话题时注入上下文。
- **素材的新鲜度**由 Cron Job 保证，定期抓取外站更新。
- **外部场景触发**（如不同群聊、不同事件）由 Webhook 负责，自动切换活跃人格。

## 三、组件职责

### 3.1 MCP Server — 动态人格引擎

**核心作用**：在对话进行中，根据当前上下文**实时检索并注入**相关 lore。

**暴露的工具（Tools）**：

| 工具名 | 输入 | 输出 | 触发时机 |
|--------|------|------|----------|
| `search_lore` | `character`, `query` | 最相关的 3-5 条 lore 卡片 | 用户提到角色关系、剧情、台词时 |
| `build_prefill` | `character`, `context` | 拼接好的轻量 prefill JSON | 会话初始化、人格切换时 |
| `list_personas` | - | 可用角色列表 | 用户问「你能扮演谁？」时 |
| `switch_persona` | `character`, `scene` | 切换确认 + 新 prefill | Webhook 或用户显式要求切换时 |

**为什么用 MCP Server 而不是改源码**：
- Hermes 原生支持 MCP，模型可以在对话中**自主决定**何时调用这些工具。
- 完全零侵入，不需要碰 `run_agent.py` 或 conversation loop。
- 一个 MCP server 可以被多个 Hermes Profile 共用。

---

### 3.2 Cron Job — 素材维护流水线

**核心作用**：让原始素材保持「半自动更新」，避免 HoYoLAB 或官方剧情更新后，本地文档沦为旧版本。

**任务清单**：

| 任务 | 频率 | 动作 |
|------|------|------|
| `wiki-sync` | 每周一次 | 检查目标角色的 HoYoLAB WIKI 是否有更新 |
| `lore-diff` | 每周一次 | 对比新旧素材，生成 `diff-report.md` |
| `git-commit` | 每周一次 | 将自动抓取的更新提交到本地仓库 |
